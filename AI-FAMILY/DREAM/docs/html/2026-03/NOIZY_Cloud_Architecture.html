Section 02 · System Design

The Three-Layer NOIZY Stack

These tools are not competitors. They are three layers of the same architecture. Remove any one of them and something specific breaks. The key is knowing which tool lives at which layer — and never asking one to do the job of another.

01
Layer One · Creator Facing
User Access Layer
Mountain Duck + Google Drive

What lives here

  • Mountain Duck: virtual disk mount in actor's DAW or editor
  • Google Drive Desktop: team docs, non-technical collaborators, onboarding materials
  • AVA's voice suggestion access — pulls from Vault via Mountain Duck mount
  • Session files, project assets, recording captures

Why this combination

  • Mountain Duck is invisible inside the DAW — creator just sees a disk
  • Google Drive handles human collaboration docs without any friction
  • Zero downloads for large voice assets — Mountain Duck streams in place
  • Non-technical actors never touch a terminal or config file
⚠ Breaks if removed If Mountain Duck is removed, actors need to manually download every voice asset before use. The AVA "instant access" promise collapses. Latency enters the creative workflow. Trust in the platform drops immediately.
02
Layer Two · Editing & Production
Workflow Layer
Mountain Duck + rclone CLI

What lives here

  • Mountain Duck: live editing of voice stems and session files in DAW
  • rclone: automated versioning of completed sessions to primary storage
  • Collaboration contract file distribution — rclone pushes signed contracts to R2
  • Quality gate output routing — post-librosa validation, rclone moves to delivery bucket

Why this combination

  • Mountain Duck handles the in-session experience (native feel, low friction)
  • rclone handles the post-session automation (no human involvement required)
  • Separates the creative act from the logistics act — neither tool tries to do both
  • rclone dedupe keeps the Vault clean — no duplicate consent keys
⚠ Breaks if removed If rclone is removed from the workflow layer, post-session versioning becomes manual. Quality gate outputs pile up locally on the M2 Ultra. The Royalty Ledger backup falls behind. The automated pipeline stalls.
03
Layer Three · Platform Backbone
Infrastructure Layer
rclone headless daemon

What lives here

  • rclone daemon: continuous routing of assets between M2 Ultra and Cloudflare R2
  • Encrypted nightly backup of SQLite Royalty Ledger (rclone crypt + R2)
  • AI training dataset pipeline: rclone filters + transfers approved training batches
  • HVS Vault integrity checks: rclone check verifies no tampering between snapshots
  • VSI partnership delivery: rclone pushes licensed output bundles to partner buckets

Why rclone (not RcloneView) at this layer

  • RcloneView GUI is for single-machine power users — not server-side automation
  • headless rclone runs as a daemon, called from Python MCPs directly
  • Full bash/Python scripting: the DreamChamber MCP can trigger rclone operations via subprocess
  • No GUI dependency = no process management complexity on M2 Ultra
⚠ Breaks if removed If the rclone daemon is removed, the entire automated pipeline requires human intervention at every step. AI training batches don't move. Ledger backups fail silently. Partner delivery stalls. The platform becomes a creative tool, not infrastructure.
04
Layer Four · Sovereign Object Storage
Storage Backend
Cloudflare R2 (Primary)

Why Cloudflare R2, not S3

  • Zero egress fees — every byte served to creators is free outbound
  • S3-compatible API — rclone supports it natively, Mountain Duck mounts it
  • Global CDN via Cloudflare Workers — sub-50ms voice asset delivery anywhere
  • GDPR-compliant jurisdictional storage — critical for HVS consent architecture

Storage structure

  • r2://noizy-vault/ — consent keys + actor voice models (encrypted)
  • r2://noizy-delivery/ — licensed output files (quality-certified, 48kHz)
  • r2://noizy-ledger/ — encrypted Royalty Ledger backups (append-only)
  • r2://noizy-training/ — approved AI training batches (consent-scoped)
⚠ At scale, M2 Ultra local storage is not the primary backend The M2 Ultra is the processing hub and consent authority. R2 is the distributed storage layer. rclone moves data between them. Mountain Duck mounts R2 buckets for creator access. This is the production architecture.
Section 03 · Brutally Honest

Where Every Tool Fails at Scale

Every tool in this stack has a ceiling. Knowing exactly where that ceiling is — before you hit it with a production workload — is the difference between a resilient architecture and a 3am incident.

🟦 Google Drive Desktop
  • 750GB per-day download cap — one busy AI training run hits this
  • Google API rate limits will throttle at millions of creator files
  • No cloud-to-cloud transfer — everything routes through local machine
  • Sync conflicts on simultaneous edits — no merge strategy for audio
  • Google terms allow scanning content — incompatible with HVS consent model
  • Cannot be scripted or automated via MCP — no programmatic control
🟣 Mountain Duck
  • Per-seat desktop license — not designed for server-side multi-tenant use
  • No automation or scheduling — purely manual workflow tool
  • Connection instability on spotty networks — kills active DAW sessions
  • Cannot orchestrate AI pipelines — there is no API or CLI
  • Cache management under heavy load requires manual intervention
  • Cannot handle cloud-to-cloud operations (partner delivery, multi-region)
🟠 rclone / RcloneView
  • FUSE mount latency — not as responsive as Mountain Duck for live editing
  • RcloneView GUI does not scale — use headless rclone daemon in production
  • macFUSE dependency — kernel extension, can conflict with macOS updates
  • Learning curve for non-technical users — never expose to creators directly
  • Single-machine orchestration ceiling — needs distributed task queue above it at millions-scale
  • Error handling requires monitoring — silent failures possible without --log-file
⚠ The Real Scale Ceiling
All three tools are desktop-class tools. They are excellent for the current phase — M2 Ultra, local SuperStack, early creator base. At millions of concurrent creators, the infrastructure layer needs to graduate to: rclone running headless as a microservice, orchestrated by a task queue (Celery / Redis), backed by Cloudflare R2 with Workers KV for consent metadata. This is not a problem for Day 1. It is a problem for Year 2. Build the rclone daemon layer now so the graduation is seamless.

Section 05 · Evolution Path

Upgrade Architecture

The current stack is correct for Day 1 through approximately 10,000 active creators. Each upgrade phase is triggered by a specific scale signal — not by a calendar date. Here is when to upgrade, what to add, and what to replace.

PHASE 1 · NOW → 10K Creators
Current Stack — Valid and Correct
M2 Ultra running rclone daemon, Mountain Duck for creator access, Cloudflare R2 as storage backend. All MCPs calling headless rclone via Python subprocess. Google Drive for team collaboration only. RcloneView GUI for debugging only — never in the critical path.
rclone daemon Mountain Duck Cloudflare R2 FastMCP (Python)
PHASE 2 · 10K → 100K Creators
Add Task Queue + Monitoring
rclone operations move from synchronous MCP calls to an async task queue (Celery + Redis). This prevents long-running transfers from blocking the MCP response loop. Add rclone log shipping to a monitoring dashboard. Vault MCP gets a read replica for consent verification at scale.
Celery + Redis rclone --log-file Grafana (rclone metrics) R2 read replica
PHASE 3 · 100K → 1M Creators
Distributed rclone + CDN Caching
Single M2 Ultra is no longer the sole processing hub. rclone runs as a distributed microservice across multiple nodes. Cloudflare Workers KV stores consent metadata for sub-millisecond verification at edge. Mountain Duck remains creator-facing; rclone headless handles all automation at this scale.
rclone microservice Cloudflare Workers KV Multi-node M2 Cloudflare Cache Rules
PHASE 4 · 1M+ Creators · Global
HVS Protocol as Global Infrastructure
rclone is no longer enough for cloud-to-cloud at global scale. The NOIZY infrastructure layer graduates to a dedicated storage orchestration service that implements the HVS protocol natively. VSI and other partners integrate via HVS SDK, not rclone configurations. The standard runs everywhere.
HVS SDK Multi-region R2 NOIZY Storage Service Global CDN mesh
CACHING · Any Phase
Voice Asset Caching Layer
Frequently accessed voice models (the top 1% of actors by casting volume) are pre-warmed in Cloudflare Cache. AVA's predictive casting engine feeds cache pre-warming decisions — if AVA predicts high demand for a voice this week, it's already at edge before the requests arrive.
Cloudflare Cache AVA predictive signal R2 → CDN warm

AVA's intelligence layer extends to storage decisions. Based on licensing demand patterns, AVA's Autonomous Deal Engine tells the storage layer where to replicate, what to pre-warm, and when to archive cold assets. The Ferris wheel generates its own storage optimisation signals.

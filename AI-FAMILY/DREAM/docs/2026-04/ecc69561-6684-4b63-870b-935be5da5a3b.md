NOIZYSTREAM: core thesis

NOIZYSTREAM should be a two-lane audio system:
	•	Studio lane: Dante as the deterministic, uncompressed, low-latency local backbone
	•	Internet lane: WebRTC as the authenticated remote edge for listen, talkback, approvals, and contribution
	•	Interop lane: AES67 only where you need cross-vendor pro-audio compatibility

That split matches what each stack is best at: Dante is built for networked AV with routing/monitoring/control across Ethernet, WebRTC is built for real-time peer media on browsers and native clients, and AES67 exists to improve interoperability across AoIP ecosystems.  ￼

Decision matrix

Stack	Latency	Audio fidelity	Implementation difficulty	Mac M2 Ultra friendliness	Studio readiness	Internet readiness
Dante	Excellent	Excellent	Medium	High	Excellent	Low to Medium
WebRTC audio	Good to variable	Good to very good	Medium to High	High	Medium	Excellent
AES67 hybrid	Excellent	Excellent	High	Medium to High	Excellent	Low to Medium

Why:
	•	Dante is explicitly positioned as a network AV platform with device routing, monitoring, clock/latency visibility, and Mac support through Dante Virtual Soundcard.  ￼
	•	WebRTC is an open standard for real-time voice/media across modern browsers and native clients, but it requires signaling/negotiation and rides real-world internet conditions, so it is less deterministic than a controlled studio LAN.  ￼
	•	AES67 is fundamentally an interoperability standard for high-performance AoIP rather than a full end-to-end product/control ecosystem.  ￼

Signal flow

Studio Mode

Mic / instrument / DAW / interface → Dante endpoint or Dante Virtual Soundcard on M2 Ultra → Dante network switch → target endpoints (record, monitor, cue, talkback, print, archive). Dante Controller is the routing layer for subscriptions, naming, latency visibility, and clock stability monitoring.  ￼

Internet Mode

Mic / DAW / interface on M2 Ultra → local capture bus → WebRTC media engine → signaling/auth service → remote listener / performer / client browser or app. WebRTC handles real-time peer media, while signaling establishes sessions and role-appropriate media paths.  ￼

Bridge Mode

Dante core → bridge node on M2 Ultra → WebRTC edge for remote approvals and contribution. This lets your studio stay deterministic while internet participants sit at the edge rather than inside the clock domain. That separation is an architecture inference, but it follows from Dante’s controlled-network design and WebRTC’s internet-native role.  ￼

Control plane

NOIZYSTREAM needs a separate control plane from the audio plane.

Control plane responsibilities
	•	session creation and teardown
	•	role assignment
	•	route templates
	•	auth and authorization
	•	health/status
	•	proof/audit records
	•	remote monitoring policy

For the Dante side, Dante Controller already covers routing/monitoring, and Dante Domain Manager or Dante Director add secure management, user access, remote control, and broader network organization. For the WebRTC side, you need your own signaling service because signaling is outside the core peer media transport.  ￼

Best NOIZYSTREAM control split
	•	Data plane: Dante streams locally; WebRTC streams remotely
	•	Control plane: NOIZYSTREAM service for sessions, roles, permissions, proof, and observability
	•	Ops plane: health dashboards, route manifests, audit log, proof bundle

That gives you a creator-facing product without rebuilding the parts Dante already solves well.  ￼

Auth model

For Studio Mode, auth should govern who can change routes, subscribe to streams, join sessions, or monitor channels, not the raw audio packets themselves. Dante Domain Manager and Dante Director are specifically about secure management, user access, and network control.  ￼

For Internet Mode, use a modern token-based model:
	•	identity provider issues short-lived JWTs
	•	signaling service validates token
	•	token claims map to session roles and permissions
	•	media permissions derive from role, not from ad hoc UI decisions

A clean permission model would be:
	•	session:create
	•	session:join
	•	stream:publish
	•	stream:subscribe
	•	stream:monitor
	•	route:modify
	•	session:record
	•	session:admin

That permission structure is an architecture recommendation, but it aligns with WebRTC’s need for application-level signaling/auth and Dante’s existing management model.  ￼

M2 Ultra local topology

Your M2 Ultra is a strong host for this design because Apple specifies up to a 24-core CPU, up to a 76-core GPU, up to 192GB unified memory, and 800GB/s memory bandwidth. That is plenty for local routing, DSP, monitoring, encoding, bridge logic, and control services on one machine.  ￼

Recommended local topology
	•	M2 Ultra Mac Studio
	•	Primary audio interface
	•	1 dedicated Ethernet path for Dante
	•	Optional second network path for internet/control
	•	Dante Virtual Soundcard on the Mac when you want the Mac to behave as a Dante endpoint
	•	Audio MIDI Setup aggregate device only if needed, with drift correction used correctly when devices are not hardware-synced

Apple explicitly notes aggregate devices need consistent sample-rate handling and drift correction when devices are not hardware synchronized.  ￼

Practical topology
	•	Thunderbolt / USB audio interface for capture/monitoring
	•	Dante Virtual Soundcard or Dante AVIO USB if needed for network audio ingress/egress
	•	managed switch for Dante subnet
	•	NOIZYSTREAM local control service on the Mac
	•	WebRTC edge service on the same Mac or adjacent node

Dante Virtual Soundcard turns the Mac into a Dante-enabled device over the wired network, and Dante AVIO USB is an option for connecting a computer to a Dante network with minimal friction.  ￼

Session roles

Use simple, explicit roles.

1. Host / Engineer
	•	creates session
	•	owns route template
	•	can arm recording and publish stems
	•	can approve contributors and listeners

2. Artist / Primary creator
	•	publishes source audio
	•	receives cue/talkback
	•	can approve collaborator access to their stream

3. Remote contributor
	•	can publish return audio and hear assigned mix-minus
	•	cannot modify routing fabric

4. Client / listener
	•	listen-only by default
	•	optional talkback if granted

5. Producer / director
	•	can monitor multiple buses
	•	can toggle talkback and notes
	•	can request route changes, but not necessarily execute them

6. Admin / ops
	•	manages infrastructure
	•	sees health, proof, and audit state
	•	no automatic right to creative content unless explicitly granted

That separation reflects the difference between media rights, routing rights, and administrative rights, which is the right design for a creator-first system. It also maps cleanly onto Dante’s managed-control features and WebRTC’s app-level authorization model.  ￼

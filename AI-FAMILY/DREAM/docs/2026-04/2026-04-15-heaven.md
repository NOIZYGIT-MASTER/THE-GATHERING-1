## 2. Input validation — FAIL

**Observed:**

- `heaven/src/index.ts` uses `await c.req.json()` in multiple POST
  handlers (`/v1/consent/check`, `/v1/consent/grant`, `/v1/consent/revoke`,
  `/v1/signup`) without a declared schema validator.
- Type assertions (`as RequestShape`) are present but these are
  compile-time only. At runtime the Worker will accept any JSON body
  and only fail later when a required field is missing or of the
  wrong type — producing 500s, not clean 400s.
- No size cap on request bodies is set at the edge. A malicious or
  accidental 10 MB payload reaches the parser before any check runs.
- The never-clause keyword classifier in `consent.ts` runs on
  untrusted `text` input. The classifier is fine, but the absence
  of an upstream length cap means a very large `text` value reaches
  the classifier and burns CPU time (Worker has `cpu_ms = 50` — a
  big payload can force a time-out rather than a clean rejection).

**Why this is a FAIL:** the consent kernel's job is to be the last
clean surface between the world and the ledger. "Crashes on bad input"
is not a clean surface.

**Action (blocking before first deploy):**

1. Add a runtime validator on every POST handler. Options, in order
   of preference:
   - `zod` (small, Worker-compatible, good errors)
   - `valibot` (even smaller)
   - hand-rolled guard functions with clear error codes
2. Add an explicit body-size limit at the Worker layer. Reject > 64 KB
   with 413 before parsing.
3. Add a max-length guard on every free-text field that feeds the
   never-clause classifier (suggest 8 KB).

## 3. Auth surface — FAIL

**Observed:**

- `/v1/health`, `/v1/stats`, `/v1/variant`, `/v1/waitlist/stats`,
  `/v1/never-clauses`, `/v1/signup` all appear to be reachable
  without authentication.
- `/v1/signup` in particular writes to `KV_GABRIEL`. An unauthenticated
  writable endpoint is a free write amplification primitive for any
  stranger on the internet.
- The CORS allowlist in `heaven/src/index.ts` includes
  `http://localhost:3000` and `http://localhost:7777`. These belong
  in a development config, not in production. Leaving them in
  production widens the allowed origin set past what the deployed
  surface actually needs.
- No rate limit is applied to `/v1/signup` or any other write endpoint.
  Cloudflare's platform-level abuse protection exists, but the
  application has no own-floor.

**Why this is a FAIL:** the charter's "external" veto class covers
"any action a stranger could receive and attribute to the architect."
Unauthenticated writes to the architect's own database are exactly
that surface — at the wrong end, inbound.

**Action (blocking before first production deploy):**

1. Either strip the localhost origins from the production CORS
   allowlist, or move them behind an `ENVIRONMENT === 'staging'`
   guard.
2. Put `/v1/signup` behind a minimum: Turnstile or hCaptcha token, or
   shared-secret header, or both. An unauthenticated insert is not
   acceptable on a production surface.
3. Add a per-IP rate limit to `/v1/signup` using a KV counter with a
   short TTL (e.g. 10 writes per IP per hour).
4. Document which endpoints are **intentionally** public (`/v1/health`,
   `/v1/never-clauses`) and which are behind auth, in the OpenAPI
   spec — so future changes are conscious, not drift.

## 4. Silent state — ADVISORY

**Observed:**

- The kernel uses both a `ledger` table (in `DB_HVS`) and an events
  model (elsewhere in the mesh, per `00-MASTER-CHARTER.md`). These
  are not yet cross-referenced: the ledger records royalty splits;
  the events table records mutations. There is no declared rule for
  which one is the source of truth for "did X happen."
- `KV_GABRIEL` is used for two distinct purposes: idempotency cache
  (consent_grant) and signup storage. Overlapping purposes in one
  KV namespace is a footgun — a cache eviction policy written for
  one purpose can silently erase the other.

**Why this is ADVISORY, not FAIL:** no data loss has been observed.
The risk is future drift, not current breakage.

**Action:**

1. Split `KV_GABRIEL` into two bindings: `KV_IDEMPOTENCY` (short TTL,
   purpose: dedupe) and `KV_SIGNUP` (long TTL, purpose: durable).
   This is a one-migration change.
2. Add a short "system of record" table to `00-MASTER-CHARTER.md`
   stating: ledger = royalty truth, events = mutation truth, D1
   `sessions`/`messages` = conversation truth.

## 5. Hardcoded endpoints — FAIL

**Observed:**

- CORS allowlist in `index.ts` contains hardcoded localhost origins
  (`http://localhost:3000`, `http://localhost:7777`). Production
  deploys should not allow localhost origins. This was flagged above
  under auth; it is also a hardcoded-endpoint finding in its own
  right because the allowlist is inline in source rather than sourced
  from `env.ENVIRONMENT` + a config map.
- Cross-reference: `RSP-NOIZY/deployments/workers/wrangler.toml`
  currently carries `REPLACE_AFTER_wrangler_d1_create_lucy` and
  `REPLACE_AFTER_wrangler_kv_namespace_create_session_hints`. Any
  deploy that sends those strings to Cloudflare will fail, and the
  `wrangler.toml` file itself encodes a non-negotiable against that
  happening. Good. But if the architect forgets to paste the real id
  after running `wrangler d1 create lucy`, a deploy will silently
  attempt with the placeholder. Recommend a preflight grep.

**Action:**

1. Move the CORS allowlist to a small config map keyed on
   `env.ENVIRONMENT`. Production gets the production origins only.
2. Add a preflight step to the deploy runbook: `grep REPLACE_AFTER
   deployments/workers/wrangler.toml` must return no matches.

## Recommended remediation order

1. **Auth surface first.** Close `/v1/signup` behind a rate limit +
   auth check. This is the biggest live-fire risk: an unauthenticated
   write endpoint is discoverable within hours of DNS going public.
2. **Input validation second.** Add `zod` (or equivalent) schemas to
   every POST. Add a 64 KB body cap. Add an 8 KB text cap.
3. **Hardcoded endpoints third.** Move CORS to env-keyed config.
   Add the `grep REPLACE_AFTER` preflight to the deploy runbook.
4. **Advisory follow-ups** (4 and 1) get logged for the next monthly
   ENGR audit per `AUDIT-CADENCE.md`.

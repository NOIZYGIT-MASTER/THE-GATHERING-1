That error is from Apple's Virtualization framework (VZErrorDomain) — a VM failed to start because its storage attachment is invalid. Common causes:

1. **Disk image path missing/moved** — the `.img`/`.dmg`/`.raw` the VM points to no longer exists at the expected path.
2. **File in use / locked** — another VM process or Finder has the disk open (VZDiskImageStorageDeviceAttachment requires exclusive access unless `readOnly: true`).
3. **Permissions** — the app lacks Full Disk Access, or the file is owned by another user.
4. **Zero-byte or corrupt image** — truncated during a crash.
5. **Unsupported format** — must be raw; qcow2/vmdk won't attach.
6. **Signed/entitlement issue** — app missing `com.apple.security.virtualization` entitlement (only matters if you're building the host app).

What's the context — which VM app (UTM, Tart, VirtualBuddy, your own Swift code)? And does the disk image file still exist at its configured path? If you share the VM config or a path, I can point at the exact fix.
